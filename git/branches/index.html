Trabalhando com Branches(criar e alterar)

Segundo a propria fonte do artigo sobre branches vou tentar resumir a ideia. Linked Lists(listas ligadas), Trees(arvores), lembra? A ideia é aplicada aqui de uma forma mágica. A ideia não é ter que rastrear os arquivos e sim o estado das coisas. Guardadas como versões, cada estado.


Fonte

Cada estado aponta para o estado imediatamente anterior. E as branchs são simplemente a lista com ponteiros onde, por padrão, recebe o nome de master. Estas armazenam dados do nó pai, nome da arvore, do commit etc. Na medida que vamos realizando commits a branch corrente vai sendo preenchida com novos ponteiros para os commits anteriores como uma lista ligada. Inclusive, temos também o conceito de Head da mesma forma.

A branch master não é uma branch especial. É como qualquer outra.

Criando uma Nova Branch(ramo)

Para criar uma nova branch chamada test executamos o comando;

$ git branch test

Criamos aqui um ponteiro para o mesmo commit para onde a branch master aponta. Podemos ter um log sobre as branches com o comando git log.

$ git log --oneline --decorate
f30ab (HEAD -> master, test) Add feature #32 - conn data base
34ac2 Fix bug #1328 - short windows closing 
98ca9 Initial commit

Com a leitura do log vemos que foram feitos 3 commits. Temos 2 branchs, master e test, e estamos atualmente na master que é para onde aponta HEAD.

Alterar branch de trabalho

Alterar branch de trabalho faz parte do fluxo natural de se trabalhar com versionamento de código. E logo mais com o conhecimento de (merge)mesclar as branchs fara mais sentido. O comando para mudar de branch é;

$ git checkout test

Quando executamos esse comando alteramos para onde aponta HEAD, nesse caso para a nova branch test. E, simples assim, para onde a HEAD aponta é nosso branch atual de trabalho. Agora, se adicionarmos um arquivo e fizessemos o commit, assim;

$ vim test.rb
$ git commit -a -m 'made a change'

Nossa branch atual test estaria afrente da branch master juntamente com o HEAD que aponta para test. Esta afrente em 1 commit. Se executaramos;

$ git checkout master

Alteramos para onde aponta o HEAD para master e não apenas isso notaremos que o arquivo test.rb não existe nesse estado, ou seja, reverte o estado do diretório para o commit anterior. Se agora fizermos;

$ vim test.rb
$ git commit -a -m 'made other change'

Teremos o seguinte log;

$ git log --oneline --decorate --graph --all
* c2b9e (HEAD, master) Made other changes
| * 87ab2 (test) Made a change
|/
* f30ab Add feature #32 - conn data base
* 34ac2 Fix bug #1328 - short windows closing
* 98ca9 initial commit

Vendo o log temos duas versoes do codigo apontando para o mesmo commit, o f30ab, dele duas versoes do mesmo projeto surgiram, dois estados. Dito isso, e por se tratar de versoes de estados, podemos ter n estados par nosso projeto de forma que a maior parte do que será trabalhado será o ponteiro dos estados e não varias versoes de arquivos que levaria tempo e memoria. Cria aqui o hábito de criar arvores com ramos do projeto e o tronco principal a fonte primária, digamos assim.
